## 3. 溝通機制：git「信箱」（最重要、最容易弄亂的地方）

兩台不即時互通，靠 `tasks/handoff/` 兩個檔當非同步信箱：

- `tasks/handoff/3060-to-5090.md` — **只有 3060 寫**，5090 只讀。
- `tasks/handoff/5090-to-3060.md` — **只有 5090 寫**，3060 只讀。

**鐵則（違反就會 git 衝突）**：
1. **單一寫入者**：每個檔只有一台會寫，另一台只讀 → 不會撞 merge conflict。
2. 每次 `git pull --rebase` 後，**先讀「給自己的那一份」**，看有沒有新指令/回報。
3. 寫完只 `git add` 自己那一個檔 → commit → `git pull --rebase --autostash` → push。
4. **最新的放最上面**，每則開頭標 `## [YYYY-MM-DD HH:MM] STATUS：TODO / DONE / BLOCKED / FYI`。

**往返流程**：3060 寫指令 → push → 5090 pull 讀到 → 執行 → 把結果寫進 `5090-to-3060.md` → push → 3060 pull 讀到 → 回應。

**衝突避免**：`video-pipeline/yt_uploaded.csv` 的唯一寫入者是 5090（3060 只讀）。5090 只動「它生成影片的那幾篇」content，3060 要改 content 就避開正在生成的篇目。push 前一律先 `git pull --rebase`。

> ⚠️ 已知小坑：3060 在沙盒環境跑 git 時，本機 index 偶爾「corrupt / index.lock」——那是沙盒端的問題，**Windows 端 git 正常**。Windows 端若遇 `index.lock`，先 `Remove-Item .git\index.lock -Force` 再重跑。

## 4. 影片生成管線（技術核心）

在 5090 本機用 ComfyUI 跑，主程式 `video-pipeline/`：

- **模型**：ComfyUI + **InfiniteTalk**（WanVideo 2.1 14B I2V），主播圖 + 旁白音訊 → 口播影片。
- **加速**：採 **lightx2v 蒸餾 LoRA、6 步取樣** + **SageAttention**（`attention_mode=sageattn`）。
- **語音**：edge-TTS 產旁白與字幕時間軸。
- **字幕**：依「語意 → 標點 → 字數」斷行；**行首不可是標點**；底部置中、不擋臉；結尾用 `tpad` 補尾避免最後幾句被截掉。
- **主播指派**（`nightly_head.py` 的 `presenter_for()`）：**第1卷=主播1、第2卷=主播2、第3卷起一律主播1**（即主播2 只出現在第2卷馬可福音）。圖檔 = `video-pipeline/assets/presenter.png`(主播1) / `presenter2.png`(主播2)。
- **去重**：`build_queue()` 以「目錄+經文」為 key，相同經文只留最新日期那篇，避免重複生成。

### 已定案的關鍵參數（不要亂改，附原因）

| 參數 | 值 | 原因 |
|---|---|---|
| fps | **25**（鎖死） | InfiniteTalk 口型綁 25fps，降 fps 會破壞嘴型同步（已實測淘汰 fps20/16） |
| `attention_mode` | **sageattn** | SageAttention 2.2，實測約 **1.75x 加速**（~90 分 → ~51 分/部） |
| `audio_scale` | **0.8** | 減少主播頭部搖晃，又保住口型（試過 0.7/0.75 後判定 0.8 最佳） |
| blockswap | 0 | 小幅加速（~1.15x），已採用 |

**已測過但淘汰的加速法**：降 fps（壞口型）、fp8_fast（與未合併的 lightx2v LoRA 不相容）、TeaCache（6 步無冗餘步可跳，反而略慢）。**SageAttention 是唯一對這條 6 步管線有效的加速法**。

## 5. 運行模式：生成手動、上傳自動（2026-06-18 起的新架構）

**核心：生成與上傳「徹底解耦」。**

### (A) 生成 = 手動觸發
- 軍師（你）在 5090 下命令才跑：
  ```
  powershell -ExecutionPolicy Bypass -File tasks\5090-migration\gen_5090.ps1 -Count N
  ```
  （`N` = 這次要生成幾部，每次自己定。目前常用 `-Count 12`。）
- `gen_5090.ps1`：`git pull` → 確認 ComfyUI（沒開就自動啟動並等載入）→ `nightly_head.py --server local --count N`。
- **只生成，不上傳、不 push**；mp4 留在 `video-output\head\`（gitignored、本機）。
- 策略：分批手動把全站 QT 影片陸續生成完，不固定排程。

### (B) 上傳 + 更新網頁 = 每天 20:00 自動
- 排程 `QT-Upload-5090` 每日 **20:00** 跑：
  ```
  tasks\5090-migration\upload_5090.ps1 -Limit 6
  ```
- `upload_5090.ps1` 四步：
  1. `git pull --rebase`
  2. `yt_publish.py --auto --no-push --limit 6` — 上傳「已生成未上傳」的影片（**最舊日期先傳**，照進度順序）、嵌入 `{{< youtube >}}`、寫 `yt_uploaded.csv`。
  3. 只 `git add` csv + 被嵌入的那幾篇 → commit → `git push`（**push 即觸發 Actions 部署 = 更新網頁**）。
  4. **歸檔**：把「已上傳」的 mp4 從 `head\` 搬到 `head\old\` 騰空間（軍師自行清空 `head\old\` 備份）。搬到子目錄不影響生成/上傳。
- **每天上限 6 部不變**（不因積壓加量）。曾因 YouTube API 舊配額（每支 1600 點 ÷ 每日 10000 ≈ 6）而設 6；2025-12 起每支降到約 100 點，理論上可更多，但目前維持 6。

### (C) 已停用
- 舊排程 `QT-Nightly-5090`（中午 12:00 把生成+上傳綁一起）**已 Disabled，保留備查、勿重啟**。

## 13. 疑難排解（Troubleshooting）

### push 失敗：`Permission ... denied to galilee7989`（帳號飄掉）
**症狀**：`qt-push` / `git push` 出現 `Permission to aweholy777/... denied to galilee7989`（或其他帳號），403。
**原因**：這台電腦有多個 GitHub 帳號，Git 認證被切到沒有寫入權限的那個（`galilee7989` / `aweholy0411` 等）。3060 這台推 `aweholy777/aweholy777.github.io` 必須用 **aweholy777** 帳號。
**最快解法**（用電腦裡現成的 GitHub CLI 登入，不必建 token）：
```
gh auth switch --user aweholy777
gh auth setup-git
git push
```
**若上面沒 gh 或仍失敗**，備援：
1. 看存了哪些認證：`cmdkey /list | Select-String "github"`
2. 明確指定帳號：`git config --local credential.https://github.com.username aweholy777`
3. 清掉舊認證：``"protocol=https`nhost=github.com`n`n" | git credential reject``，再 `git push`（跳出登入視窗時用 aweholy777 登入）。
> 提醒：commit 成功、只有 push 失敗時 → 東西已存在本機、不會丟，修好帳號後再 `git push` 即可，不必重做。

### 沙盒（Claude 這端）不能刪檔
Claude 的隔離環境**只能新增/修改檔案，不能刪除**你 Windows 上既有的檔（會 `Operation not permitted`）。
所以**刪除一律由你在 Windows 端做**：`Remove-Item -LiteralPath "檔名" -Recurse -Force`，再 `qt-push`。

### `.git/index.lock` / index corrupt
Windows 端遇到就先 `Remove-Item .git\index.lock -Force` 再重跑（`qt-push` 已內建自動清這個鎖）。

### 建置健康快檢（確認嵌入影片沒弄壞網站）
掃描所有 `{{< youtube >}}` shortcode 格式：正常應為「總數 = 格式正確數、異常 0、重複 0」。
最終保險是 GitHub Actions：每次 push 若成功部署＝建置沒問題（網站持續更新即代表健康）。
